1. Purpose & Scope

This document defines the complete business and functional requirements for the QBank Admin Portal — the internal-facing web application used by a Client's operations staff to manage customers, transactions, compliance, risk, and system configuration.

This BRD is the authoritative technical reference for the engineering team building the Admin Portal. It covers every screen, role, permission level, workflow, approval chain, and data requirement derived from the approved UX prototype (qbank_admin_v2.html). Any deviation from this document must be approved by the Product Office before implementation.

Who This Document Is For: Frontend engineers, backend engineers, QA engineers, architects, and the product team. This is not a user-facing document — it is an internal technical specification.

1.1 What is the Admin Portal?

The Admin Portal is the internal operations interface deployed within each QBank Client's instance. It is entirely separate from the Customer Portal. No customer ever accesses the Admin Portal. All Client staff (compliance officers, risk analysts, operations agents, system administrators) use the Admin Portal to:

  • Review and approve/reject customer onboarding applications (KYC/KYB)
  • Monitor and investigate transactions, fraud alerts, and risk events
  • Manage failed, stuck, and pending payment transactions
  • Handle customer support tickets and account management
  • File Suspicious Activity Reports (SARs) with regulators
  • Manage staff user accounts and system configuration

2. Admin Portal Architecture

2.1 Architecture Principles

🔐

Role-Driven UI

Every screen, navigation item, button, and action is rendered based on the logged-in user's role and level. No generic admin view exists.

🏗️

Single-Page Application

The admin portal is a SPA. All views are rendered client-side. Navigation does not cause full page reloads. State is maintained per session.

📋

Drawer + Modal Pattern

Detail views use a right-side slide-in drawer (not separate pages). Confirmation dialogs use centered modals. Toast notifications for feedback.

🤖

AI-Augmented Decisions

AI risk scoring, recommendations, and anomaly signals are embedded directly in review workflows — not separate screens.

📎

Immutable Audit Trail

Every admin action is logged with timestamp, actor, IP address, and target. The audit log is read-only and exportable.

⛓️

Level-Gated Actions

High-risk actions are locked behind clearance levels. Lower-level users see the action but must raise an escalation to trigger it.

3. Authentication & Session Management

3.1 Login Flow

Login Screen /login
Fields
  • Email address (text input, pre-filled as admin@qbank.com in demo)
  • Password (masked input)
  • Sign In button (triggers credential validation)
  • Inline error message: "Invalid credentials. Please try again." (shown on failure, hidden on load)
Demo Credential Hints (Development / QA only)

A collapsible panel at the bottom of the login card shows demo credentials for each role and level. Each credential card is clickable to auto-fill the email and password fields. This panel must be disabled/removed in production builds.

Behaviour
  • On valid credentials → transition to MFA screen (login screen hidden, MFA screen visible)
  • On invalid credentials → show error message inline, keep login screen visible
  • On success → store the matched user object in application state (role, level, name, initials, IP)
MFA Screen /mfa
Fields
  • 6 individual single-character digit inputs (focus auto-advances on each entry)
  • Verify Code button
  • Inline error if fewer than 6 digits entered
Behaviour
  • Auto-focus moves from digit 1 through digit 6 as user types
  • On digit 6 entry, form auto-submits (no button click required)
  • In production: validate 6-digit TOTP against the user's registered authenticator app secret
  • On success → hide MFA screen, activate App Shell, call initApp()
  • App initialization: renders sidebar, pre-renders all shared table views, starts session timer, writes login event to audit log

The top bar displays persistently: a clock icon + "Session: MM:SS" counter on the right side, a role badge (colour-coded by role), and the logged-in user's name. These elements are visible at all times while in the app shell.

4. Role Hierarchy & Level Structure

Critical Design Principle: The Admin Portal has four role departments. Within each of the three operational departments (Compliance, Risk, Operations), there are three seniority levels (L1, L2, L3). The Super Admin is a separate fourth role with a fixed Level 4. A user belongs to exactly one role and exactly one level. These are set at account creation and can only be changed by a Super Admin.

Each user's dashboard includes a Level Panel at the top, immediately below any pending escalation alerts. The Level Panel shows:

  • The user's exact title (e.g., "L2 — Compliance Officer")
  • A one-sentence description of their primary responsibilities at this level

  • "Sign out" link (red on hover)

5. Permissions Matrix

How Permissions Work: Every action in the system runs through a can(action) check. Super Admin bypasses all checks. For other roles, the check looks up the action key in the role's permission map and returns true only if the user's level is in the allowed levels array. Actions not permitted display as "locked" buttons (dashed border, padlock icon) that, when clicked, open the Escalation Modal instead of performing the action.

5.4 Super Admin Permissions

🔴 Super Admin (Level 4) — Unrestricted Access

The Super Admin bypasses all role-based permission checks. The can(action) function returns true unconditionally for the Super Admin role. The Super Admin can see and perform every action across all departments.

Exclusive Super Admin capabilities (not available to any other role):

  • Create, edit, and deactivate staff user accounts
  • Assign and change user roles and levels
  • Configure system-wide settings (currencies, payment rails, onboarding requirements)
  • Manage API keys and third-party integration credentials
  • Rotate API keys for BaaS, FX, and KYC providers
  • Configure security settings (MFA enforcement, session timeout, IP allowlist)
  • View system health dashboard (service uptime, response times, incidents)
  • The Super Admin's own account cannot be deactivated through the UI (protected)

6.1 Escalations Navigation Badge

The "Escalations" nav item shows a red numeric badge when there are pending items directed at the current user's role and level. The badge count is the sum of:

  • Simple escalations (ESC-xxx) where targetRole === currentRole AND targetLevel === currentUser.level AND status === "pending"
  • Multi-Level Requests (MLR-xxx) where the chain config role matches currentRole AND currentStage === currentUser.level AND status === "in_progress"

The badge is not shown for Super Admin (they have no escalation queue).

7. Escalation & Multi-Level Approval System

QBank's admin portal has a two-track escalation system: (1) Simple Escalations for ad-hoc situations where a lower-level user needs a higher-level user to take an action they're not cleared for; and (2) Multi-Level Requests (MLR) for predefined high-stakes actions that require a mandatory sequential approval chain.

7.1 Simple Escalations

Trigger: A user (L1 or L2) clicks a locked action button. The Escalation Modal opens. They fill in a reason and optional notes. On submit, an ESC record is created and routed to the appropriate level within the same department.
FieldDescription
ESC IDAuto-generated reference (e.g., ESC-001)
Raised ByName, level, and role of the user raising the escalation
Target LevelThe level that needs to action this (L2 or L3 within the same role)
Target RoleMust match the raiser's role department
ActionThe specific action key that requires escalation (e.g., approve_high_risk)
Action LabelHuman-readable description (e.g., "Approve Application (High Risk)")
TargetThe entity being acted upon (e.g., APP-2844)
Target NameHuman-readable target (e.g., "Lisa Park")
ReasonMandatory text field — why this escalation is being raised
NotesOptional supporting context
Statuspending → actioned / returned
TimestampISO datetime of creation
TrailAppend-only array of {who, ts, note} — audit history of escalation events

Simple Escalation Actions (for the approver):

  • Approve: The higher-level user executes the underlying action (e.g., approves the application). The ESC status is updated to actioned. The trail records the decision with notes.
  • Reject: The higher-level user rejects the escalated request. Trail updated with reason. The original item remains in its pre-escalation state.
  • Return to Requester: Opens the Return Escalation Modal. The requester is notified the escalation was returned with a reason — they can revise and re-raise.

7.2 Voluntary Forward

Any L1 or L2 user (in any department) can voluntarily forward a decision to a higher level for review even when they technically have permission to act. This is for complex edge cases or when a second opinion is needed. This triggers a forward record similar to an escalation, but does not block the original item — it is advisory.

The "Forward for Review" button appears in drawers for L1 and L2 users whenever a relevant action is available. Clicking it opens the Voluntary Forward Modal with a target level selector (L2 or L3) and a reason field.

7.3 Multi-Level Requests (MLR) — Mandatory Approval Chains

MLR actions cannot be performed by a single person regardless of level. They require a full sequential sign-off chain. Each stage must be approved before the next stage is notified. If any stage rejects, the entire MLR is rejected and the action is blocked.

13.4 Drawers

  • Drawers slide in from the right. Standard width: 520px. Wide variant (Consumer 360): 560px.

  • Drawers have a fixed header (title + subtitle + close button), a scrollable body, and a fixed action bar at the bottom.
  • Transition: cubic-bezier easing, 300ms, sliding from off-screen right to right:0.

13.5 Toast Notifications

  • Position: fixed, bottom-right, above all content.
  • Types: success (green left border) | error (red) | warning (amber) | info (navy)
  • Auto-dismiss after ~3 seconds with fade-out animation.
  • Stack vertically if multiple toasts appear simultaneously.

13.6 Tables

  • Header row: navy background (#1a3c6e), white text, 11–12px uppercase with letter-spacing.
  • Alternating row background: even rows use --light (#f0f4ff).
  • Row hover: #e8eeff.
  • Reference codes (IDs, refs, IPs): Consolas monospace, accent blue, bold.

13.7 Collapsible Drawer Sections

All sections within the Review Drawer use the collapsible pattern: a header row with title and a chevron (▼/▲). The first section (AI Risk Intelligence) defaults to open. All others default to closed. Click the header to toggle. The body animates show/hide with display:block/none (no CSS transition — instant show/hide).

16. Glossary

TermDefinition
Admin PortalThe internal-facing web application used by a Client's staff to manage QBank operations
ClientThe financial institution that has licensed and deployed QBank software
CustomerAn end user of the Client's banking service — never accesses the Admin Portal
RBACRole-Based Access Control — the permission model governing what each admin user can see and do

MLRMulti-Level Request — a mandatory sequential approval chain for predefined high-stakes actions
Voluntary ForwardProactively forwarding a decision to a higher level for second opinion, even when the current user has permission to act
Consumer 360The wide slide-in drawer showing a customer's full profile, accounts, transaction history, and notes
KYCKnow Your Customer — identity verification process for retail customers
KYBKnow Your Business — verification process for business customers
SARSuspicious Activity Report — regulatory filing submitted to the FCA (UK) or relevant authority
PEPPolitically Exposed Person — individual with a prominent public role, requiring enhanced due diligence
AMLAnti-Money Laundering — regulatory framework and screening processes to detect financial crime
ReconciliationThe process of verifying that the sum of all customer currency ledgers matches the actual omnibus bank account balance held by the banking partner
Omnibus AccountA pooled bank account held by QBank's banking infrastructure partner, containing all customer funds in a given currency
FCAFinancial Conduct Authority — UK financial regulator. SAR submissions go to the FCA.
TOTPTime-based One-Time Password — the algorithm used for MFA (RFC 6238, compatible with Google Authenticator, Authy, etc.)
SPASingle-Page Application — the Admin Portal architecture where navigation does not trigger full page reloads
Top-upAn admin-initiated credit to a customer's ledger, used for corrections, refunds, or adjustments. Requires L2+ clearance.
Velocity BreachA risk alert triggered when a customer's transaction volume or frequency exceeds configured thresholds within a time window
Risk ScoreA 0–100 integer produced by the AI Risk Engine indicating the likelihood of fraud or compliance risk for an applicant or transaction. <40 = Low, 40–69 = Medium, 70–84 = High, 85+ = Critical.
